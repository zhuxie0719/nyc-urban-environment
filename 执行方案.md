## 4. 指标体系（ZIP 统一、面向推荐）

### 4.1 环境核心评分（你已有）
统一成 0–100（越高越好）：
- `air_quality_score`（空气更好 → 分更高）
- `noise_score`（更安静 → 分更高）
- `green_score`（更绿 → 分更高）
- `environment_score`（加权综合分）

建议保留并明确权重（可在 UI 中展示）：
\[
environment\_score = 0.45 \cdot air + 0.35 \cdot noise + 0.20 \cdot green
\]

### 4.2 新增：热风险维度（HVI）
HVI 原始通常是 1–5（越大越危险），建议转换成 0–100 的“越高越好”：
- `heat_safety_score`：把 HVI 反向映射到 0–100（例如 1→100，5→0，可用线性映射）

### 4.3 新增：公共设施覆盖指标（至少做 2 个）
为了“应用型决策”，建议做两类：

**(A) 覆盖密度类（ZIP 内设施数量或密度）**
- `cooling_site_count`（降温点数量）
- `restroom_count`（公厕数量）
- `fountain_count`（饮水点数量）
- `park_area_ratio`（公园面积占比，或公园面积/ZIP面积）

**(B) 可达性近似类（ZIP 到最近设施的距离）**
若暂时不做路网路由，可用“到最近设施点的直线距离”作为近似：
- `nearest_cooling_site_km`
- `nearest_restroom_km`
- `nearest_fountain_km`

然后把距离反向映射成 0–100（越近越好）形成：
- `facility_access_score`（公共设施可达性分）

### 4.4 应用型总分（可调权重）
给用户一个“我在意什么”的可调权重：
- `fit_score`（前端实时计算，不一定写死到 JSON）
\[
fit = w_{env}\cdot environment + w_{heat}\cdot heat\_safety + w_{fac}\cdot facility\_access
\]
其中 `w_env + w_heat + w_fac = 1`（UI 用滑块/预设）。

## 5. 可视化设计与交互（重点：应用与可解释）

### 5.1 主视图：ZIP 着色地图（推荐/筛选）
- **形式**：Choropleth（按 `fit_score` 或 `environment_score` 着色）
- **交互**：
  - 权重滑块（环境/热风险/设施可达性）
  - 阈值过滤（Top N 或 >= 某分）
  - 悬停：tooltip 显示各项子分
  - 单击：锁定区域（你已实现），用于“对比与解释”

### 5.2 新奇点：双变量机会地图（Bivariate “机会象限”）
把它定位为“用户筛选器”，不是学术分析：
- **形式**：2D 色标（横轴：环境分，纵轴：热安全分 或 设施可达性分）
- **交互**：
  - 点击色标象限直接过滤（例如“高环境 + 低热风险”）
  - 显示过滤后 ZIP 数量 + 推荐 Top10



### 5.4 解释器（Why this ZIP）
锁定后生成“推荐理由/风险提示”：
- **形式**：文本卡片 + 指标条形图
- **内容**（自动生成规则）：
  - 优势 Top2：例如“噪音分高于全市均值 +15”
  - 风险 Top1：例如“热安全分偏低，建议优先选择附近 Cooling Center 覆盖更高区域”

### 5.5 公共设施分布可视化（新增主内容）
避免地铁重复，重点做“热浪与日常需求”：

**(A) 设施点位图层（多图层开关）**
- 图层：Cooling Centers / Drinking Fountains / Public Restrooms / Parks
- 交互：
  - 图层开关（checkbox）
  - 点位聚合（zoom out 时用 cluster，减少遮挡）
  - 点击点位：弹出名称、开放时间/可用性字段（若数据提供）

**(B) 覆盖盲区模式（服务空白）**
- 形式：选择一种设施（如 Cooling Centers）后，地图以 ZIP 显示“最近距离/可达性分”
- 交互：
  - 阈值滑块：只高亮“距离最近设施 > X km”的 ZIP
  - 与热风险叠加：同时高亮“高热风险 + 设施远”的双重劣势区（非常应用）

### 5.6 场景模式（Scenario Presets）
让老师一眼看出“应用价值”：
- **怕热浪**：提高 `w_heat`，自动打开 Cooling Centers 图层，切换到“盲区模式”
- **需要安静**：提高 `w_env`，并在解释器强调噪音
- **喜欢户外**：提高 `w_env` 中绿化权重（或提高 `green_score` 权重），打开 Parks 图层
- **日常便利**：提高 `w_fac`，打开 Restrooms + Fountains

## 6. 数据处理流程（ZIP 口径）

### 6.1 统一空间键
- 主键：`zip_code`（5 位）
- 所有设施点位（Cooling/Fountain/Restroom）需要：
  - 点 → 映射到 ZIP（point-in-polygon），得到 `zip_code`

### 6.2 HVI 口径说明（关键）
HVI 常见口径是社区统计区（如 UHF/NTA），可能不直接给 ZIP：
- 方案 A（推荐，最快交付）：将 HVI 区域的中心点/边界与 ZIP 做空间叠加，得到 ZIP 的平均/众数 HVI
- 方案 B（更精确）：按 ZIP 的人口权重叠加（需要 ACS 人口栅格/tract，工作量大，可作为“未来工作”写入）

### 6.3 公共设施可达性近似（无需后端）
用直线距离近似：
- 计算 ZIP 多边形质心到最近设施点的距离
- 再反向映射为 0–100 分
> 在说明文档中明确“近似假设”，并写未来可升级为路网路由（ORS/OSRM）。

## 9. 风险与兜底

- **HVI 口径不匹配 ZIP**：先做空间叠加近似（方案 A），并在 Limitations 说明  
- **设施点位数据字段不齐**：只保留经纬度 + 名称也足够完成覆盖/距离/盲区功能  
- **性能问题（点位多）**：用点聚合（cluster）或只显示当前视窗内点位
